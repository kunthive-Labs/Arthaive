§ 01 — What this is

An India-first answer to a paywalled question.

If you want to ask “show me every Series A in fintech from Q1 2026,” you either pay thousands for Tracxn/Crunchbase or scrape the news by hand for hours. Arthaive is the free, structured, third option.

Arthaive extracts structured funding data from the articles that journalists already write, normalizes it, and makes it searchable — without charging for access and without losing the trail back to the source. It is built for founders benchmarking valuations, investors scouting pipeline, journalists looking up histories, and analysts pulling aggregate trends.

Not a news site

We don't write articles. We extract structured data from the articles other people write.

Not a Crunchbase clone

We focus on India and stay free. We don't chase global coverage.

Not unverified data

Every record links back to its source article. Wrong number? Click through and check.

§ 02 — The eight load-bearing rules

Every feature is measured against these.

From PROJECT_REFERENCE.md §5. If a decision drifts from one of these, it gets pushed back. They are not aspirations — they are gates.

P1

Source-backed data

Every funding event in the public dataset has at least one source URL. No sourceless records.

P2

Canonical identity

One startup = one row, one investor = one row. Aliases ("Sequoia India" → "Peak XV") map to canonical names so dedup works.

P3

Normalized structure

Startups, rounds, investors, sources are separate tables joined by foreign keys — not one fat spreadsheet.

P4

Incremental updates

The pipeline runs daily and adds new records. It never rebuilds the dataset from scratch.

P5

Human verification

Extracted records pass through a review queue. High-confidence ones auto-approve; the rest wait for an admin.

P6

Explainable analytics

Every chart traces to its rows. If you can't show the deals behind a number, the chart doesn't ship.

P7

Cost efficiency

AI calls are reserved for what regex can't handle and high-value summaries. We don't Claude-every-line.

P8

Honest data labeling

We disclose coverage gaps. Charts show date ranges. The UI says what's verified vs. estimated.

The dual-source pattern

The key file is lib/db/deals.ts. getDeals(filters) tries Supabase first and falls back to the static data/funding-data.ts (1,696 deals) when the DB is empty or unconfigured. This is how the migration happens gradually — the frontend works whether or not the live DB is populated.

§ 04 — Data model

A normalized graph, not a spreadsheet.

sources and investors feed deals; review_queue gates what enters; alias tables resolve identity; pipeline_jobs records every run.

Core · the verified dataset

deals

Single source of truth for confirmed funding events. company, amount_inr/usd, stage, sectors[], investors[], deal_date, source_url. Public reads filter on record_status='verified'.

Core · provenance

sources

One row per URL ever processed. url is UNIQUE — the URL-dedup key. Stores reliability_tier and a 5,000-char raw snapshot, so the evidence survives even if the article goes dead.

Core · the gate

review_queue

Pipeline output awaiting humans. Keeps the full raw_extracted_data (jsonb), a suggested_company, a match_confidence, and a status: pending → approved / rejected / merged.

Entity · identity

startup_aliases
investor_aliases

The truth table for "these strings mean the same entity." (company, alias_name, alias_type) — former_name, alternate_spelling, short_name, brand_name. Trigram GIN index for fuzzy lookups.

Operational · audit

pipeline_jobs

One row per run. Counts fetched / filtered / extracted / auto-approved / flagged, plus run_status and error log. Powers the admin "Pipeline" health page.

User-facing

profiles · alerts
bookmarks · saved_searches

Per-user features: display name & bio, saved deals, saved queries, watchlists, and email-digest alerts on new matching deals.

Deliberate trade-offs, not oversights

deals.company is text, not a FK to a startups table — the alias table already gives canonical identity (P2). deals.investors is a text array, not a join table. record_status lives on deals so public reads stay one simple filter. All reversible if they start hurting.

§ 05 — The data collection pipeline

How a news article becomes a row.

The most important chapter. A Python CLI you run on a cron — nine steps from a source sitemap to a verified deal, every run additive thanks to URL-level dedup.

Discovery

pipeline/discovery.py

Walks the source's daily-partitioned sitemap, pulls URLs newer than --since, applies a coarse keyword pre-filter (slugs like raises, secures, series).

URL-level dedup

dedup.py · is_url_seen

Skips any URL already in sources. This is what makes repeated runs safe and additive (P4).

Fetch

fetcher.py + wayback.py

Downloads with httpx, extracts the body via CSS selectors. Dead link? Falls back to the Wayback Machine. Body truncated to 5,000 chars to bound token cost.

AI extraction

extractor.py · Claude Haiku

Returns strict JSON — company, amount, currency, stage, sectors, investors, date, and a self-rated confidence. Non-funding articles return is_funding_event=false and are dropped. Responses cached in SQLite by sha256(url+body) — re-runs are free.

Currency normalization

currency.py

Converts amount + currency into both amount_inr and amount_usd at a fixed rate (83.5). Good enough for round-level charts; the only file to change when historical rates arrive.

Entity resolution

entity_resolver.py

Maps every company & investor name to its canonical form — exact alias hit, then rapidfuzz. (See §06.)

Deal-level dedup

dedup.py · find_duplicate_deal

Same company + deal_date within ±30 days + amount_inr within ±10% → treated as a duplicate (likely a second publisher) and routed to review with a hint.

Routing

run.py · _route_record

Auto-approve only if all hold: confidence ≥ 0.80, canonical company exists, date non-null, amount > 0, no duplicate. Pass → deals (verified, instantly public). Else → review_queue.

Job log

queue.py · log_job_run

One row to pipeline_jobs with all counts and run status — the data behind the admin health dashboard.

§ 06 — Entity resolution

"Sequoia India" and "Peak XV" are the same investor.

Sequoia rebranded to Peak XV in 2023. Store them as separate rows and search misses three-quarters of their deals, the leaderboard splits, and histories lie. Resolution is the fix for principle P2.

Two halves: the alias tables (startup_aliases, investor_aliases) are the source of truth for "these strings mean the same entity," seeded day-one with ~30 known cases. The resolver (entity_resolver.py) runs three checks in order — exact alias hit → exact canonical hit → fuzzy match via rapidfuzz token_sort_ratio against every canonical name, alias, and previously-verified deal.

SCORE ≥ 92
Auto-canonicalize
Replace the name before insert. False positives are rare.
SCORE 75–91
Suggest & review
Attach as suggested_company, route to the queue. A human confirms.
SCORE < 92, under 75
Treat as new
No good match → probably a new entity. Use the name as-is.
The closed loop — every variant is a one-time cost

Aliases grow from three sources: the day-one seed, the admin entity manager, and auto-alias on approval (Phase 5.4) — when a reviewer confirms "Sequoia India" → "Peak XV Partners" once, the next pipeline run resolves it without their help. Every name learned is remembered.

Conscious limits: the alias system collapses parent/subsidiary, treats global Sequoia & India Peak XV as one, and can't yet distinguish a person who is both angel and founder. When simple stops being enough, it layers on rather than rebuilds — and at ~50k+ entities the fuzzy match moves into Postgres via the existing trigram index.

§ 07 — The admin workflow

Where humans keep the data clean.

The pipeline auto-approves the confident extractions; everything ambiguous piles up in review_queue. The admin console at /admin is where principle P5 lives. Access is gated by the ADMIN_EMAILS env var.

/admin

Dashboard — counts, recent activity, links

/admin/review

The queue — extractions awaiting a decision

/admin/entities

Canonical names & aliases manager

/admin/sources

Publisher registry & reliability tiers

/admin/pipeline

Run logs, counts, errors, status

/admin/import

Bulk CSV import of historical deals

/admin/export

Pull the verified dataset as CSV / JSON

audit

Every action stamps reviewed_by & reviewed_at

The core loop — four actions on a review item
ActionWhat happens to the database
ApproveInsert a verified row into deals; mark queue item approved. (Phase 5.4: also auto-write the alias if the raw name differed.)
RejectMark rejected with a note. Stays for audit, never enters deals.
MergeMark merged — a duplicate of an existing deal; optionally append the new source URL.
Needs more infoMark needs_more_info — unclear article or undisclosed amount; come back later.
A typical admin day

Check /admin/pipeline (was last night green?) → work /admin/review (source article is one click away) → approve / reject / merge → alias any new name variants. A diligent admin clears ~50 items in 30 minutes once entity resolution starts pre-suggesting correctly.

§ 10 — Run it locally

The whole stack on your laptop in ~15 minutes.

Prereqs: Node 20+, Python 3.10+, a free Supabase project, and an Anthropic API key (pipeline only — the frontend runs without it on the static fallback).

  1. Clone & install. Frontend deps with npm install; pipeline deps in a venv with pip install -r pipeline/requirements.txt.
  2. Environment. Copy .env.example → .env.local and fill in the Supabase URL + anon + service-role keys, ADMIN_EMAILS, and ANTHROPIC_API_KEY.
  3. Database. Run migrations 001 … 018 in the Supabase SQL editor (or supabase db push), then seed with seed_aliases.sql.
  4. Frontend. npm run dev → open localhost:3000. Empty DB? It falls back to the 1,696-deal static file, so the site is fully browsable.
  5. Pipeline. Dry-run first, then a real run (below). Check sources, review_queue & pipeline_jobs in Supabase.
  6. Admin. Sign up with an email in ADMIN_EMAILS, then visit /admin.
# dry run — no DB writes, no AI calls
python -m pipeline.run --source entrackr --since 2026-05-01 --dry-run --limit 5

# the real thing
python -m pipeline.run --source entrackr --since 2026-05-20 --limit 10

§ 11 — Glossary

The vocabulary.

The domain terms used across the codebase and these docs — the shorthand that makes the rest legible.

Canonical name
The single authoritative name for an entity. "Peak XV Partners" is canonical; "Sequoia India" is an alias of it.
Alias
A name variant that maps to a canonical name. Stored in startup_aliases / investor_aliases.
Record status
Gates public visibility: verified (public), needs_review, rejected, merged. The site filters on verified.
Source
A row in sources — one per unique URL. Every deal links to at least one (P1).
Reliability tier
tier_1 direct reporting (Entrackr, Inc42) · tier_2 reliable but rehashed · tier_3 discovery hints only.
Confidence
Claude's self-rated 0–1 score per extraction. Drives auto-approval. Distinct from the resolver's 0–100 match score.
Auto-approval threshold
Confidence ≥ 0.80 plus guards: canonical company, non-null date, amount > 0, no duplicate.
Match confidence
rapidfuzz token_sort_ratio, 0–100. ≥92 auto · 75–91 suggest · <75 new entity.
URL-level dedup
Skipping an article whose URL is already in sources — how the pipeline stays safely re-runnable.
Deal-level dedup
Detecting a re-reported round: same company, ±10% amount, ±30 days.
Wayback fallback
When a live URL is dead, try archive.org's snapshot.
Extraction cache
.extractor_cache.sqlite — every Claude response keyed by sha256(url+body). Re-runs hit the cache, not the API.
Coverage notice
The disclaimer telling users what date range & dataset the charts cover (P8).
Undisclosed amount
Round announced without a figure — stored amount_inr=0, shown as "Undisclosed," never auto-approved.
RLS
Row-Level Security — Postgres filtering that enforces "public read on verified, admin write only."

Source-backed by design.

Every funding event in Arthaive traces to the article it came from. That single rule — P1 — shapes the schema, the pipeline, the review queue, and the analytics. This guide is the explanation; PROJECT_REFERENCE.md is the spec, and it wins any conflict.
